 22: /**
 23:  * Red-Black tree-based implementation of Map. This class guarantees
 24:  * that the map will be in both ascending key order and ascending
 25:  * value order, sorted according to the natural order for the key's
 26:  * and value's classes.<p>
 27:  *
 28:  * This Map is intended for applications that need to be able to look
 29:  * up a key-value pairing by either key or value, and need to do so
 30:  * with equal efficiency.<p>
 31:  *
 32:  * While that goal could be accomplished by taking a pair of TreeMaps
 33:  * and redirecting requests to the appropriate TreeMap (e.g.,
 34:  * containsKey would be directed to the TreeMap that maps values to
 35:  * keys, containsValue would be directed to the TreeMap that maps keys
 36:  * to values), there are problems with that implementation,
 37:  * particularly when trying to keep the two TreeMaps synchronized with
 38:  * each other. And if the data contained in the TreeMaps is large, the
 39:  * cost of redundant storage becomes significant.<p>
 40:  *
 41:  * This solution keeps the data properly synchronized and minimizes
 42:  * the data storage. The red-black algorithm is based on TreeMap's,
 43:  * but has been modified to simultaneously map a tree node by key and
 44:  * by value. This doubles the cost of put operations (but so does
 45:  * using two TreeMaps), and nearly doubles the cost of remove
 46:  * operations (there is a savings in that the lookup of the node to be
 47:  * removed only has to be performed once). And since only one node
 48:  * contains the key and value, storage is significantly less than that
 49:  * required by two TreeMaps.<p>
 50:  *
 51:  * There are some limitations placed on data kept in this Map. The
 52:  * biggest one is this:<p>
 53:  *
 54:  * When performing a put operation, neither the key nor the value may
 55:  * already exist in the Map. In the java.util Map implementations
 56:  * (HashMap, TreeMap), you can perform a put with an already mapped
 57:  * key, and neither cares about duplicate values at all ... but this
 58:  * implementation's put method with throw an IllegalArgumentException
 59:  * if either the key or the value is already in the Map.<p>
 60:  *
 61:  * Obviously, that same restriction (and consequence of failing to
 62:  * heed that restriction) applies to the putAll method.<p>
 63:  *
 64:  * The Map.Entry instances returned by the appropriate methods will
 65:  * not allow setValue() and will throw an
 66:  * UnsupportedOperationException on attempts to call that method.<p>
 67:  *
 68:  * New methods are added to take advantage of the fact that values are
 69:  * kept sorted independently of their keys:<p>
 70:  *
 71:  * Object getKeyForValue(Object value) is the opposite of get; it
 72:  * takes a value and returns its key, if any.<p>
 73:  *
 74:  * Object removeValue(Object value) finds and removes the specified
 75:  * value and returns the now un-used key.<p>
 76:  *
 77:  * Set entrySetByValue() returns the Map.Entry's in a Set whose
 78:  * iterator will iterate over the Map.Entry's in ascending order by
 79:  * their corresponding values.<p>
 80:  *
 81:  * Set keySetByValue() returns the keys in a Set whose iterator will
 82:  * iterate over the keys in ascending order by their corresponding
 83:  * values.<p>
 84:  *
 85:  * Collection valuesByValue() returns the values in a Collection whose
 86:  * iterator will iterate over the values in ascending order.<p>
 87:  *
 88:  * @author Marc Johnson (mjohnson at apache dot org)
 89:  */

521:     /**
522:      * get the next larger node from the specified node
523:      *
524:      * @param node the node to be searched from
525:      * @param index _KEY or _VALUE
526:      *
527:      * @return the specified node
528:      */
529:     static Node nextGreater(Node node, int index)
530:     {
531:         Node rval = null;
532: 
533:         if (node == null)
534:         {
535:             rval = null;
536:         }
537:         else if (node.getRight(index) != null)
538:         {
539: 
540:             // everything to the node's right is larger. The least of
541:             // the right node's descendents is the next larger node
542:             rval = leastNode(node.getRight(index), index);
543:         }
544:         else
545:         {
546: 
547:             // traverse up our ancestry until we find an ancestor that
548:             // is null or one whose left child is our ancestor. If we
549:             // find a null, then this node IS the largest node in the
550:             // tree, and there is no greater node. Otherwise, we are
551:             // the largest node in the subtree on that ancestor's left
552:             // ... and that ancestor is the next greatest node
553:             Node parent = node.getParent(index);
554:             Node child  = node;
555: 
556:             while ((parent != null) && (child == parent.getRight(index)))
557:             {
558:                 child  = parent;
559:                 parent = parent.getParent(index);
560:             }
561:             rval = parent;
562:         }
563:         return rval;
564:     }
