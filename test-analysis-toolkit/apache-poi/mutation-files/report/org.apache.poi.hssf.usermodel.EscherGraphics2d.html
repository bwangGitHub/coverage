 39: /**
 40:  * Translates Graphics2d calls into escher calls.  The translation is lossy so
 41:  * many features are not supported and some just aren't implemented yet.  If
 42:  * in doubt test the specific calls you wish to make. Graphics calls are
 43:  * always drawn into an EscherGroup so one will need to be created.
 44:  * <p>
 45:  * <b>Important:</b>
 46:  * <blockquote>
 47:  * One important concept worth considering is that of font size.  One of the
 48:  * difficulties in converting Graphics calls into escher drawing calls is that
 49:  * Excel does not have the concept of absolute pixel positions.  It measures
 50:  * it's cell widths in 'characters' and the cell heights in points.
 51:  * Unfortunately it's not defined exactly what a type of character it's
 52:  * measuring.  Presumably this is due to the fact that the Excel will be
 53:  * using different fonts on different platforms or even within the same
 54:  * platform.
 55:  * <p>
 56:  * Because of this constraint you have to calculate the verticalPointsPerPixel.
 57:  * This the amount the font should be scaled by when
 58:  * you issue commands such as drawString().  A good way to calculate this
 59:  * is to use the follow formula:
 60:  * <p>
 61:  * <pre>
 62:  *      multipler = groupHeightInPoints / heightOfGroup
 63:  * </pre>
 64:  * <p>
 65:  * The height of the group is calculated fairly simply by calculating the
 66:  * difference between the y coordinates of the bounding box of the shape.  The
 67:  * height of the group can be calculated by using a convenience called
 68:  * <code>HSSFClientAnchor.getAnchorHeightInPoints()</code>.
 69:  * </blockquote>
 70:  *
 71:  * @author Glen Stampoultzis (glens at apache.org)
 72:  */

330:     /**
331:      * Fills a (closed) polygon, as defined by a pair of arrays, which
332:      *  hold the <i>x</i> and <i>y</i> coordinates.
333:      * <p>
334:      * This draws the polygon, with <code>nPoint</code> line segments.
335:      * The first <code>nPoint&nbsp;-&nbsp;1</code> line segments are
336:      *  drawn between sequential points
337:      *  (<code>xPoints[i],yPoints[i],xPoints[i+1],yPoints[i+1]</code>).
338:      * The final line segment is a closing one, from the last point to
339:      *  the first (assuming they are different).
340:      * <p>
341:      * The area inside of the polygon is defined by using an
342:      *  even-odd fill rule (also known as the alternating rule), and
343:      *  the area inside of it is filled.
344:      * @param xPoints array of the <code>x</code> coordinates.
345:      * @param yPoints array of the <code>y</code> coordinates.
346:      * @param nPoints the total number of points in the polygon.
347:      * @see   java.awt.Graphics#drawPolygon(int[], int[], int)
348:      */
349:     public void fillPolygon(int xPoints[], int yPoints[], int nPoints)
