 28: /**
 29:  * <p>Abstract superclass for the convenience classes {@link
 30:  * SummaryInformation} and {@link DocumentSummaryInformation}.</p>
 31:  *
 32:  * <p>The motivation behind this class is quite nasty if you look
 33:  * behind the scenes, but it serves the application programmer well by
 34:  * providing him with the easy-to-use {@link SummaryInformation} and
 35:  * {@link DocumentSummaryInformation} classes. When parsing the data a
 36:  * property set stream consists of (possibly coming from an {@link
 37:  * java.io.InputStream}) we want to read and process each byte only
 38:  * once. Since we don't know in advance which kind of property set we
 39:  * have, we can expect only the most general {@link
 40:  * PropertySet}. Creating a special subclass should be as easy as
 41:  * calling the special subclass' constructor and pass the general
 42:  * {@link PropertySet} in. To make things easy internally, the special
 43:  * class just holds a reference to the general {@link PropertySet} and
 44:  * delegates all method calls to it.</p>
 45:  *
 46:  * <p>A cleaner implementation would have been like this: The {@link
 47:  * PropertySetFactory} parses the stream data into some internal
 48:  * object first.  Then it finds out whether the stream is a {@link
 49:  * SummaryInformation}, a {@link DocumentSummaryInformation} or a
 50:  * general {@link PropertySet}.  However, the current implementation
 51:  * went the other way round historically: the convenience classes came
 52:  * only late to my mind.</p>
 53:  *
 54:  * @author Rainer Klute <a
 55:  * href="mailto:klute@rainer-klute.de">&lt;klute@rainer-klute.de&gt;</a>
 56:  */
